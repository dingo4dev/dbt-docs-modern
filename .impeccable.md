# Design Context

## Users

**Primary Audience:** Data engineers, analytics engineers, and data analysts working with dbt projects. They are technical professionals who need to explore complex data transformations, understand model dependencies, debug pipelines, and document data architecture.

**Context of Use:**
- Often working across multiple devices (desktop at workstation, tablet/tablet during meetings, mobile on-the-go)
- Need rapid access to information during development, debugging, or stakeholder presentations
- May be exploring unfamiliar codebases or verifying their own work
- Value both comprehensive detail and quick scanning

**Job to Be Done:** Understand data lineage, model relationships, and column-level metadata with minimal friction so they can make informed decisions about data architecture transformations.

---

## Brand Personality

**Voice:** Technical precision without arrogance — confident, clear, and concise

**Tone:** Professional yet approachable, respecting the user's expertise while making complex information digestible

**Three-Word Personality:**
1. **Competent** — Works reliably, handles complexity gracefully
2. **Clear** — Information architecture reduces cognitive load
3. **Polished** — Thoughtful details that inspire confidence

**Emotional Goals:**
- **Calm & Focused:** Clean interface that reduces visual noise, helps users concentrate
- **Efficient & Productive:** Fast search, instant filtering, minimal clicks to reach information
- **Delighted & Inspired:** Moments of polish (smooth transitions, thoughtful micro-interactions) that make the experience enjoyable

---

## Aesthetic Direction

**Visual Tone:** Technical & Professional with warmth

**References:** Data products like Snowflake, dbt Labs, Airbyte — analytics-focused interfaces that balance information density with visual clarity

**Anti-References:**
- Generic admin templates (churn, lack identity)
- Overly playful/cartoonish designs (undermines technical credibility)
- Dense, text-heavy legacy documentation tools (feels outdated, intimidating)

**Theme:** Warm technicalism — approachable professional design that feels modern but not trendy

## Design Principles

### 1. Calm Efficiency First
Reduce cognitive load through clear visual hierarchy, generous spacing, and purposeful color use. Every element should earn its place. Prioritize essential information over decorative elements.

**Why:** Users work with complex data systems — the interface should be the calm part of their workflow.

**How to apply:**
- Use whitespace deliberately to create breathing room
- Color should convey meaning, not decoration
- Progressive disclosure: show overview first, details on demand
- Avoid visual clutter; remove elements that don't serve clear purpose

### 2. Technical Precision
Design should reflect the structured, logical nature of data systems. Consistent spacing, aligned grids, and intentional typography choices convey reliability.

**Why:** Data professionals trust precise, systematic interfaces. Sloppy design undermines confidence in the tool.

**How to apply:**
- Maintain consistent 4px/8px spacing scale
- Align elements to grids (visible or invisible)
- Use mono fonts for code/identifiers, sans for UI
- Respect existing design tokens (OKLCH colors, radii, transitions)

### 3. Accessibility is Non-Negotiable
Screen reader optimization is foundational. Semantic HTML, ARIA labels, and keyboard navigation are not add-ons.

**Why:** Analytics engineers may use screen readers or keyboard-only navigation. Accessibility benefits everyone.

**How to apply:**
- Use semantic HTML elements (`<nav>`, `<main>`, `<button>`)
- Provide ARIA labels for interactive elements
- Ensure keyboard navigation works for all features
- Don't rely on color alone to convey information
- Test with screen readers when building complex features

### 4. Mobile-First Performance
The interface must work beautifully on small screens. Touch targets, responsive layouts, and optimized performance are baseline expectations.

**Why:** Users check documentation on mobile devices during meetings or while away from their desks.

**How to apply:**
- Minimum 44px touch targets
- Responsive breakpoints: <640px (mobile), 640-1024px (tablet), ≥1024px (desktop)
- Test on real devices, not just browser resize
- Optimize bundle size and render performance
- Progressive enhancement: core functionality works without JavaScript

### 5. Delight in Details
Moments of polish — smooth transitions, thoughtful hover states, micro-interactions — transform functional tools into enjoyable experiences.

**Why:** Small delights build trust and make the tool memorable without compromising efficiency.

**How to apply:**
- Use 180ms cubic-bezier transitions for smooth feel
- Add hover states that provide feedback
- Celebrate user actions (copied to clipboard, added to favorites)
- Animate changes, don't snap
- Balance animation subtlety with visibility

## Technical Constraints

**Single-File Architecture:** All code must bundle into one HTML file (~200KB). Svelte 4 (not 5) for stability with vite-plugin-singlefile.

**Performance:** Large projects (>1000 models) may slow rendering. Prioritize efficient client-side filtering and virtual scrolling where appropriate.

**Browser Support:** Modern browsers with ES6+ support. Mobile Safari requires special attention for layout bugs.

---

## Design System Foundations

**Colors:** OKLCH warm neutral palette with amber accent — already defined in `src/app.css`

**Typography:**
- UI: Plus Jakarta Sans (professional, highly readable)
- Code: JetBrains Mono (excellent legibility for data/SQL)

**Spacing:** 4px base unit, 8px major scale — Tailwind defaults

**Radii:** 0.75rem (12px) for cards, 0.375rem (6px) for small elements

**Transitions:** 180ms `cubic-bezier(0.25, 1, 0.5, 1)` for smooth, responsive feel

**Components:** Reusable patterns defined in `src/app.css` (badges, cards, buttons, inputs, tags)
